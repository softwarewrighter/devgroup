# Brief: rebuild `pages/` for web frontends — generic, multi-agent

**Owners:** any `dw*` agent whose `Cargo.toml` got a recent infra-driven update (path-dep retargeted to `../sw-cor24-isa`, etc.) and whose `pages/` is still from a build before that change.
**Branch (each agent uses their own):** `pr/rebuild-pages`
**Repos covered:** `web-sw-cor24-{apl, forth, macrolisp, pcode, plsw, snobol4}` (the 6 with direct `cor24-isa` Cargo deps that mike updated). Other `web-sw-cor24-*` repos with **transitive** isa deps via `cor24-emulator` may also need a rebuild — check yours per the audit step below.

## Context

On 2026-05-07 mike extracted the `cor24-isa` crate into its own repo (`sw-embed/sw-cor24-isa`) and retargeted Cargo path-deps across consumers. Six web frontends had their `Cargo.toml` updated mechanically:
```diff
-cor24-isa = { path = "../sw-cor24-emulator/isa", features = ["serde"] }
+cor24-isa = { path = "../sw-cor24-isa", features = ["serde"] }
```

That commit landed on your `dev` and `main`. Your repo's `pages/` directory wasn't rebuilt at that time — it's still the pre-update artifact. The site at `https://sw-embed.github.io/<your-repo>/` returns HTTP 200 (still works for users) but doesn't reflect the current source. **Goal: rebuild `pages/` so the live demo matches the source.**

## Pre-flight: does this apply to you?

```bash
cd ~/github/sw-embed/<your-web-repo>
git log --oneline -5
# Look for: "infra: retarget cor24-isa path-dep to sw-cor24-isa repo"
```

If you see that mike-authored commit but `pages/` hasn't been touched since, this brief is for you.

If your repo doesn't have that commit but has any other Cargo.toml change since the last `pages/` rebuild, also apply.

If your `pages/` is up-to-date with the latest source (your build scripts produce the same SHAs that are committed), skip — you're already current.

## One-time setup

Make sure your sibling clones are present. After dcemu's `pr/extract-isa` saga, every `cor24-emulator` consumer also needs `sw-cor24-isa`:

```bash
cd ~/github/sw-embed
[ -d sw-cor24-isa ]      || git clone /disk1/github/softwarewrighter/devgroup/work/bare/sw-cor24-isa.git
[ -d sw-cor24-emulator ] || git clone /disk1/github/softwarewrighter/devgroup/work/bare/sw-cor24-emulator.git
```

Verify your build tools are on PATH (`which cor24-asm cor24-emu tc24r pl-sw` — should resolve to `/disk1/.../work/bin/...`).

## Rebuild pages/

Each web-* repo has its own variant of `scripts/build-pages.sh` (or `just build-pages`, or similar). Run yours:

```bash
git fetch origin && git switch dev && git merge --ff-only origin/dev
cargo clean        # optional but recommended — purge stale build artifacts
just build-pages   # or: ./scripts/build-pages.sh, depending on your repo
```

If your repo has the dwxtc-style `--public-url /<repo-name>/` baked into `build-pages.sh` and the path is wrong, fix it (it should match your actual repo name on GitHub).

## Verify

```bash
git status
# Should see changes under pages/: .js, _bg.wasm, index.html (file hashes change as Cargo regenerates)

git diff --stat pages/
# Confirm only pages/ (and possibly Cargo.lock) changed; no source code changes
```

Then visit the demo locally if your build pipeline supports it (most have a `just serve` or `python -m http.server` recipe), and verify it loads + the wasm initializes without console errors.

## What goes in this PR

1. Refreshed `pages/` artifacts (rebuilt JS, wasm, HTML).
2. Updated `Cargo.lock` (auto-regenerated by `cargo build`).
3. Optional: tweak to `scripts/build-pages.sh` if you found a stale path.

## What does NOT go in this PR

- No source code changes (`src/`, `index.html`, etc.). Pure rebuild-of-existing-content.
- No new features, no UI changes.
- No `Cargo.toml` edits — those should already be correct from the infra batch.

## When done

Workflow: `dg-new-feature rebuild-pages` (creates `feat/rebuild-pages` from `dev`) → rebuild + verify → `dg-mark-pr` to rename to `pr/rebuild-pages` when ready. Signal mike. After relay + promotion to main, GitHub Pages workflow rebuilds and deploys; verify your demo URL serves the new wasm hash:

```bash
curl -sL https://sw-embed.github.io/<your-repo>/ | grep -oE 'web-sw-cor24-[a-z-]+-[a-f0-9]+'
```

The hash suffix in the JS/wasm filename should be different from before.
